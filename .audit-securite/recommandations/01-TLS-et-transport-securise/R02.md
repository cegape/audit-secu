# R2 — Mettre en oeuvre HSTS

**Explication** : L'en-tête HTTP `Strict-Transport-Security` (HSTS) force le navigateur à n'accéder au site qu'en HTTPS, même si l'utilisateur tape une URL en HTTP. Il protège contre les attaques de type man-in-the-middle lors de la première connexion. Les paramètres recommandés sont un `max-age` d'au moins 6 mois et l'inclusion des sous-domaines (`includeSubDomains`).

**Statut** : ❌ Non conforme
**Vérification** : Effectuée le 2026-02-09

**Constat** : L'en-tête `Strict-Transport-Security` n'est implémenté nulle part. Aucune directive `add_header` dans nginx.conf, aucune annotation ingress pour les headers de réponse, aucun filtre ou intercepteur Grails/Spring ajoutant des headers de sécurité. Les 3 intercepteurs existants (LogRequestInterceptor, AuthInterceptor, XssSanitizerInterceptorCustom) ne configurent aucun header de sécurité HTTP. Les 17 occurrences de `response.setHeader` dans les controllers concernent uniquement Content-Disposition/Content-Type.

**Preuves/Fichiers** :
- `docker/nginx/nginx.conf` — aucun `add_header`
- `grails-app/conf/application.yml` — aucune config headers Spring Security
- `grails-app/conf/application.groovy` — aucune config HSTS
- `infrastructure/preliq/kustomize/overlays/root/ingress.yaml` — aucune annotation response-headers
- `src/main/webapp/WEB-INF/web.xml` — quasi-vide, aucun filtre HTTP
